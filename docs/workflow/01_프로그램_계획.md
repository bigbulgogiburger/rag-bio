# 01. 프로그램 계획

## 1) 프로그램 목표
Bio-Rad CS 조직이 연구자 질의에 대해 **근거 기반 판정(맞음/틀림/조건부)**과 **표준화된 고객 커뮤니케이션**을 빠르게 제공할 수 있는 MVP를 6개 스프린트 내 출시한다.

## 2) 범위(Release Scope)
### MVP 포함
- 문의 접수 + 문서 업로드(PDF/Word)
- OCR 분기 처리(스캔 PDF)
- chunk(1000) 기반 임베딩/벡터 인덱싱
- RAG 검색 + 검증 + 답변 생성
- 출처(문서/페이지/chunk) 표기
- 이메일/고객메시지 초안 생성
- 승인 후 발송(Human-in-the-loop)

### MVP 제외
- 완전 자동 발송(무승인)
- 다국어 전체 지원
- 제품군 전 영역 확장

## 3) 스프린트 운영안 (2주 단위)
- Sprint 1: 플랫폼 기초 + 문의/업로드 기본
- Sprint 2: OCR/청킹/임베딩/인덱싱 안정화
- Sprint 3: RAG 검색/검증(판정엔진)
- Sprint 4: 답변/출처 + 커뮤니케이션 초안
- Sprint 5: 승인플로우 + 보안/운영 관측
- Sprint 6: 베타 운영 + 튜닝 + 릴리즈

## 4) 팀 역할
- PM: 백로그 우선순위, KPI/리스크 관리
- FE: 운영 UI, 결과/승인 UX
- BE: 도메인/오케스트레이션/API
- AI Eng: 프롬프트/RAG/평가셋 품질
- QA: 테스트 전략/회귀
- DevOps: CI/CD/관측성/배포

## 5) KPI (가설 포함)
- 응답속도: P95 8초 이내(인덱싱 완료 질의 기준)
- 출처 포함률: 95%+
- 판정 정확도: 초기 80% → 베타 88%+
- CS 재작성률: 30% 이하
- 1차 응답 리드타임: 기존 대비 40% 단축

## 6) 리스크 및 완화
- OCR 품질 편차 → confidence 기반 재처리 큐
- 환각/단정 답변 → 근거 부족 시 보수 응답
- 문서 버전 충돌 → version policy + 최신우선 규칙
- 오발송 위험 → 승인 없는 발송 차단 + 감사로그

## 7) 게이트(진입/종료 기준)
### Sprint 시작(DoR)
- 요구사항/AC 명확
- API 계약 초안 존재
- 테스트 관점 정의

### Sprint 종료(DoD)
- 코드/테스트/문서 반영
- FE/BE 빌드 성공
- 관측포인트(로그/메트릭) 반영

## 8) 의사결정 로그(필수 기록)
- 판정 기준 변경
- RAG 파라미터(top-k/threshold/chunk)
- 발송 승인 정책 변경
