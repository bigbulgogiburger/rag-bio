# 22. UI/UX 한글화 + 현대화 리디자인 PRD

## 1) 문서 개요
- 프로젝트: Bio-Rad CS RAG 멀티에이전트 앱
- 작성일: 2026-02-13
- 목적: 기존 화면의 텍스트/디자인/UX를 전면 개선하여, **한글 중심의 명확한 사용성**과 **현대적인 제품 경험**을 제공한다.

---

## 2) 배경 및 문제 정의
현재 UI는 기능 검증 중심으로 구현되어 다음 한계가 있다.
1. 화면 항목명/문구가 영문+한글 혼재되어 인지부하가 큼
2. 레이아웃/컴포넌트 스타일이 프로토타입 수준으로 일관성이 부족함
3. 사용자 흐름(문의 생성 → 상태 확인 → 분석 → 답변 생성/승인/발송)에서 피드백/가이드가 약함

---

## 3) 목표
### 비즈니스 목표
- 데모/내부 운영 단계에서 “실사용 가능한 완성도”로 품질 상향

### 제품 목표
- 모든 주요 UI 텍스트를 한글로 표준화
- 디자인 시스템 기반의 현대적 UI 적용
- 핵심 업무 흐름의 UX 마찰 최소화

### 성공 지표 (MVP)
- 주요 화면 한글화율: 100%
- 핵심 흐름(문의 생성~조회) 성공률: 95%+
- 사용자 피드백(가독성/직관성): 기존 대비 개선

---

## 4) 범위
### In-Scope
- 프론트엔드 전체 텍스트 한글화
- 레이아웃/타이포/컬러/간격/버튼/입력폼 스타일 리디자인
- 상태/오류/로딩/성공 피드백 UX 개선
- 핵심 화면 정보 구조(IA) 개선

### Out-of-Scope (이번 PRD)
- 백엔드 비즈니스 로직 변경
- 신규 외부 채널 연동 기능 자체
- 모바일 네이티브 앱 개발

---

## 5) 대상 사용자
- CS 담당자(문의 접수/검토/발송)
- 기술 검토자(근거/판정 확인)
- 운영 담당자(상태 모니터링)

---

## 6) 정보 구조(IA) 개편안
1. **문의 접수**
   - 문의 작성, 채널 선택, 문서 첨부
2. **진행 상태**
   - 문의 상태 조회, 인덱싱 상태, 재처리
3. **분석/답변 생성**
   - 근거 검색 + 판정, 답변 톤/채널 선택
4. **검토/승인/발송**
   - 리뷰 코멘트, 승인, 발송, 감사로그

---

## 7) UX 원칙
1. **한눈에 이해**: 용어를 업무 언어(한글)로 통일
2. **즉시 피드백**: 클릭 후 로딩/성공/실패 상태를 명확히 제공
3. **실수 방지**: 위험 동작(승인/발송)은 확인 및 상태 조건 표시
4. **진행 흐름 가시화**: 현재 단계와 다음 행동을 명확히 안내
5. **일관성**: 버튼/입력/배지/메시지 패턴 통일

---

## 8) UI 디자인 가이드 (MVP)
### 비주얼 톤
- 키워드: Clean / Trustworthy / Operational
- 카드 기반 레이아웃 + 충분한 여백

### 컬러
- Primary: 청록/블루 계열 (신뢰감)
- 상태색:
  - 성공: Green
  - 진행중/주의: Amber
  - 실패: Red
  - 정보: Blue/Gray

### 타이포그래피
- 본문 가독성 우선(한글 최적 폰트)
- 제목/본문/보조텍스트 계층 명확화

### 컴포넌트
- 버튼: 기본/보조/위험 타입
- 입력폼: 라벨/힌트/에러메시지 일관
- 배지: 상태(RECEIVED, PARSED, SENT 등) 시각화
- 토스트/인라인 알림: 액션 결과 즉시 전달

---

## 9) 한글화 정책
1. 모든 사용자 노출 텍스트는 한글 기본
2. 영문 상태코드는 내부값으로 유지하되, 화면에는 한글 라벨 매핑
   - 예: RECEIVED → 접수됨, SENT → 발송완료
3. 전문 용어는 한국어 우선 + 필요 시 괄호 병기
4. 빈 상태/오류 문구도 한국어로 일관 관리

---

## 10) 핵심 사용자 시나리오
### 시나리오 A: 문의 생성
- 사용자가 문의/채널/문서를 입력
- 제출 시 로딩 표시 → 성공 토스트 + 생성된 문의 ID 자동 입력

### 시나리오 B: 상태 조회
- 문의 ID 조회 시 상태 카드 표시
- 인덱싱 요약/문서 상태를 한글 라벨로 제공

### 시나리오 C: 분석 및 답변
- 분석 질문 입력 후 판정/근거 확인
- 톤/채널 선택 후 답변 초안 생성
- 신뢰도/리스크 경고 문구를 시각적으로 강조

### 시나리오 D: 승인/발송
- 리뷰 → 승인 → 발송 흐름에서 단계별 버튼 활성화 조건 제공
- 실패 원인은 사용자 친화 문구로 안내

---

## 11) 기능 요구사항
### FR-1 텍스트 한글화
- 모든 버튼/라벨/placeholder/상태/오류/성공 문구 한글화

### FR-2 상태 UI 표준화
- 상태값 표시 컴포넌트 도입(배지/설명)

### FR-3 피드백 UX 강화
- 로딩 스피너, 성공/실패 토스트, 인라인 에러 영역 제공

### FR-4 접근성
- 키보드 네비게이션, 대비, 라벨 연결(aria) 기본 준수

### FR-5 반응형
- 최소 1280 기준 데스크톱 최적화 + 768 이상 태블릿 대응

---

## 12) 비기능 요구사항
- 성능: 기존 대비 렌더 체감 성능 저하 없음
- 유지보수성: 공통 컴포넌트/디자인 토큰 기반 관리
- 일관성: 화면 간 UI 규칙 재사용

---

## 13) 기술 구현 방향
- 프론트 기준:
  - 디자인 토큰(색상/간격/타이포) 정리
  - 공통 UI 컴포넌트 계층 재구성
  - 텍스트 상수 분리(한글 문구 테이블)
- 백엔드:
  - API 스키마 변경 최소화
  - 프론트 표시용 상태 라벨 매핑은 FE 우선 처리

---

## 14) 단계별 실행 계획 (제안)
### Sprint U1 (1주)
- 한글화 1차 + 핵심 컴포넌트 리디자인
- 문의 생성/조회 화면 우선 개선

### Sprint U2 (1주)
- 분석/답변/승인/발송 영역 UX 개선
- 상태 배지, 경고/오류 피드백 고도화

### Sprint U3 (0.5~1주)
- 접근성/반응형/디테일 폴리싱
- QA 및 사용자 피드백 반영

---

## 15) 수용 기준 (Acceptance Criteria)
1. 메인 워크플로우 화면 텍스트 100% 한글화
2. 문의 생성/조회/분석/답변 생성이 디자인 변경 후 정상 동작
3. 오류/로딩/성공 피드백이 모든 주요 액션에 존재
4. 상태값이 사용자 친화 한글 라벨로 표시
5. 브라우저 E2E 스모크 테스트 통과 + 스크린샷 증빙

---

## 16) 리스크 및 대응
- 리스크: 디자인 변경 중 기능 회귀
  - 대응: 기존 통합 테스트 + E2E 스모크 병행
- 리스크: 한글 용어 일관성 부족
  - 대응: 용어집(Glossary) 운영
- 리스크: 작업 범위 확장
  - 대응: 화면 우선순위(문의/조회/분석/답변) 고정

---

## 17) 즉시 실행 백로그(다음 액션)
1. `frontend` 텍스트 리소스 테이블 도입(한글 키 정리)
2. 문의 생성/조회 카드 UI부터 리디자인 적용
3. 상태 배지 컴포넌트 추가(RECEIVED~SENT 한글 매핑)
4. 성공/오류 토스트 공통 컴포넌트 적용
5. Playwright 캡처로 변경 전/후 비교 리포트 작성
